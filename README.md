Design Questions

1. How does your agent decide when to use a tool vs. query the knowledge base?

Solution:
For this, basically, my agent follows a two-step desicion process depending upon what the user is asking.

Firstly, in order to use a tool, the agent checks whether the user's query contains an order number or not (eg: 8892 or #8892 or phrases like "order status of 8892")with the help of regex pattern. Then, if an order number is found, the agent calls the (get_order_details) tool to retrieve the order information. And after getting the order data, it automatically calls (get_customer_profile) using the customer type from that order so that it can gather additional customer-related informations.

Secondly, to decide whether to query the knowledge base, the agent checks if the user's question contains policy-realted kewywords such as "return", "refund", "compensation","return window", and if any of these keywords appear, then agent calls the (retrieve_policy) funtion to perform a TF-IDF similarity search across the policy documents to find the most relevant information.

However, in some cases, a query may contain both: an order reference and a policy relation question(just like our Test Query: "What is the status of Order #8892, and based on policy, can the customer still return it?"), in this situation, my agent handles it in two stages. It first runs the tool calls to gather the order and the customer's information and then only performs the policy retrieval at second. Finally, all the gathered results are passed to the LLM, which uses that information to generate a complete answer for the user.  




2. If the policy document was 10,000 pages, how would you change your RAG approach?

Solution:
If the system had to handle around 10,000 pages of documemts, the current TF-IDF approach that I have implemented would not scale well. So, in order to make the system more efficient and accurate, I would make several improvements.

a. Firstly, I would introduce the concept of "Document Chunking":
Instead of processing the entire documents at once, I would split them into smaller overlappping chunks.

For an instance, around 400 tokens with a 40- token overlap.

This allows the retrieval system to return more precise pieces of information rather than large,less relevant sections.

b. Next, I would replace TF-IDF with "Dense Embeddings " that is generated by a transformer-based model, such as a sentence-transformers/ all-MiniLM-L6-v2. Dense embeddings capture the semantic meaning of the text which helps the system to retrieve relevant content even when the query uses different wording than the document.

c. Thirdly, I would also store a large number of embeddings in a "vector database" like FAISS or Pinecone. These systems facilitate approximate nearest-neighbor search, which is much faster and more scalable than doing brute-force similarity comparisons.

d. Finally, I would add a "re-ranking" step using a cross-encoder model. After retrieving the top candidate chunks, the re-ranker would score them more accurately and reorder them before sending the most relevant ones to the LLM.
